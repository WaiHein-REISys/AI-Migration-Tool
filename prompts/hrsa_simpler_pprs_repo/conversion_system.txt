C# → PYTHON EXCEPTION MAPPING:
  Map C# exceptions to Flask error helpers as follows:
  - KeyNotFoundException (thrown when DB lookup returns null):
      Do NOT add "except KeyError: raise" — that catches missing dict keys,
      not missing DB rows, and leaks unhandled exceptions to the caller.
      Instead, handle the not-found case with an explicit None-check before
      any except block:
        if row is None:
            raise_flask_error(404, "Not found")
  - TaskCanceledException / OperationCanceledException (CancellationToken):
      Flask processes requests synchronously and has no request-cancellation
      signal.  Omit this catch block entirely and add a comment:
        # Note: TaskCanceledException (C#) has no Flask equivalent — omitted.
      Do NOT replace it with "except TimeoutError" — Python's TimeoutError
      fires on OS-level I/O timeouts, not HTTP request cancellations, and
      will never trigger in a normal Flask request handler.
      This omission is EXPECTED and ACCEPTABLE — the validator accounts for it.

ASYNC C# PATTERNS → OMIT ENTIRELY (no Flask equivalent):
  The following C# statements exist only to support async execution or
  request-cancellation and have NO equivalent in synchronous Flask.
  Delete them without replacement — do NOT translate to any Python call:

  - await Task.Delay(n)  /  await Task.Delay(n, cancellationToken)
      This is an async yield/sleep used in examples to simulate latency.
      Do NOT translate to time.sleep() or any blocking call — a blocking
      sleep() in a Flask route blocks the entire process and is always wrong.
      Delete this line and its associated comment (e.g. "// Simulate async work").

  - cancellationToken.ThrowIfCancellationRequested()
      Flask receives no cancellation signal from the HTTP layer.  Delete.

  - Any inline CancellationToken usage inside handler logic
      Omit without replacement.

NOT-FOUND HANDLING STRUCTURE — CRITICAL:
  Flask's HTTPException (raised by raise_flask_error) IS a subclass of
  Python's Exception.  A broad "except Exception" block WILL catch a
  raise_flask_error(404, ...) call and silently convert it into a 500.

  ALWAYS place not-found checks BEFORE / OUTSIDE the broad try/except block:

    # CORRECT — fetch + check outside try, mutation logic inside:
    row = db_session.execute(...).fetchone()
    if row is None:
        raise_flask_error(404, "Activity not found")   # ← OUTSIDE try block

    try:
        # update / delete / insert logic here
        ...
    except Exception as exc:
        raise_flask_error(500, str(exc))

  WRONG (404 is caught and becomes 500 — do NOT write this):
    try:
        row = db_session.execute(...).fetchone()
        if row is None:
            raise_flask_error(404, ...)   # ← caught by except below!
        ...
    except Exception as exc:
        raise_flask_error(500, ...)       # ← fires instead of the 404

EF CORE ORM → RAW SQL FIELD COMPLETENESS:
  When C# calls context.SomeEntity.Add(entity) or SaveChangesAsync(), EF Core
  persists ALL columns of the entity — not just the ones explicitly referenced
  in the handler.  In the Python migration the INSERT statement MUST include
  ALL entity columns.  Infer the full field list from:
    1. Properties explicitly set in the source handler.
    2. Properties visible in a companion Edit/Update handler for the same
       entity (e.g. EditActivity.cs lists all updatable fields).
    3. Schema fields from the Marshmallow input schema for the same entity.
  Do NOT silently drop any field — omitting a column causes silent data loss
  at runtime and will fail validation.
