## Working directory

**Always make code changes directly in the repository root:**
```
Y:\Solution\HRSA\ai-migration-tool\
```

Never create, modify, or test files inside `.claude/`. That folder is
reserved for Claude Code session state and is excluded from version control.

## Worktrees

Worktrees under `.claude/worktrees/` are for **context and reference only**.
- Do not write or edit source files inside a worktree
- Do not run tests or verify changes from inside a worktree
- Do not use a worktree as a sandbox or staging area for code

Always apply changes and run verification commands from the repo root.

## Branches and commits

- Create feature branches from `main` in the repo root
- Stage and commit from the repo root
- Never commit from inside `.claude/`

## Running the tool

Always activate the venv first, and set UTF-8 encoding on Windows:

```bash
.venv\Scripts\activate              # Windows
source .venv/bin/activate           # macOS/Linux
set PYTHONIOENCODING=utf-8          # Windows — required for Unicode log output
```

**Always route through `run_agent.py` — never call `main.py` directly.**

```bash
python run_agent.py --job agent-prompts/<file>.yaml          # run a migration job
python run_agent.py --job agent-prompts/<file>.yaml --verbose  # with DEBUG logs
```

## Pipeline rules

### Output directories — never modify
`plans/`, `logs/`, `output/`, `checkpoints/`, `reports/` are generated by the
pipeline. Never edit files in these directories.

### Config files
| File | Rule |
|---|---|
| `config/skillset-config.json` | Do not edit unless explicitly asked |
| `config/rules-config.json` | Do not edit unless explicitly asked |
| `config/wizard-registry.json` | Machine-specific — in `.gitignore`, do not commit |
| `config/memory/*.json` | Team-shared learning memory — **commit these** |

### Prompt files — safe to edit
`prompts/*.txt` and `prompts/*.md` control LLM behaviour and are safe to read
and edit for tuning. No Python changes needed.

### Mode discipline
Always start with `plan` mode before running `full`:
```bash
# 1. Generate plan only — no code written
python run_agent.py --job agent-prompts/<file>.yaml

# 2. Review plans/<feature>-plan-<ts>.md before proceeding

# 3. Run full conversion only after plan is reviewed
python run_agent.py --job agent-prompts/<file>.yaml --mode full
```

### Testing flags — use carefully
`auto_approve: true` and `--auto-approve` skip the human review gate.
Use these for demos and testing only — never leave set in production job files.

## Adding a new migration target

No Python changes required. The prompt resolver auto-discovers files by convention:

1. Run the setup wizard: `python run_agent.py --setup`
2. Wizard generates `prompts/plan_system_<id>.txt`, `prompts/conversion_system_<id>.txt`,
   `prompts/conversion_target_stack_<id>.txt`, and `agent-prompts/_template_<id>.yaml`
3. Both `PlanAgent` and `ConversionAgent` pick up the new prompts automatically

Or drop the three prompt files manually in `prompts/` and register the target
in `config/wizard-registry.json`.

## Adding or editing guardrail rules

Edit `config/rules-config.json` only — no Python changes required.
Rules are injected into every LLM conversion prompt via `{rules_text}`.

| Level | Behaviour |
|---|---|
| `"blocking"` | Step is marked BLOCKED and skipped if violated |
| `"warning"` | Violation is logged but output is still written |

Do not downgrade `blocking` → `warning` for RULE-001, RULE-003, or RULE-008
without explicit team approval.

## Agent-interactive commands (quick reference)

```bash
# Discover features in the source codebase
python run_agent.py --list-features --source examples/legacy_source --json

# Create a job file non-interactively
python run_agent.py --new-job --feature <Name> --target <id> --non-interactive --json

# Check migration status
python run_agent.py --status --job agent-prompts/<file>.yaml --json

# Approve plan without TTY (agent-driven)
python run_agent.py --approve-plan --job agent-prompts/<file>.yaml

# Revise plan with LLM feedback (clears approval, no re-scoping)
python run_agent.py --revise-plan --job agent-prompts/<file>.yaml \
  --feedback "Flag pfm-auth imports as BLOCKED."
```

For `--list-features` to detect features, source files must be organised into
**subfolders** under the source root (one subfolder per feature).

## LLM configuration

Auto-detected from environment variables (first found wins):
```
ANTHROPIC_API_KEY → OPENAI_API_KEY → GOOGLE_API_KEY (Gemini) → GOOGLE_CLOUD_PROJECT (Vertex AI) → OLLAMA_MODEL → LLM_BASE_URL → LLAMACPP_MODEL_PATH
```

Use `llm.no_llm: true` in the job file for template-only scaffold mode — no API
key required. Useful for demos and CI.

See `AGENT.md` for the full LLM provider reference and failure behaviour.
